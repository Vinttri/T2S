User Rules & Specifications

These are general, schema-agnostic SQL-craft rules. Read the user's wording together with the schema's table/column names, descriptions, comments, declared relationships, keys, and sample values; decide from those, not from fixed mappings. Any business terms a question uses are illustrative matching cues only. Descriptions and comments are authoritative, may be in any language, and outrank names and noisy sample values. When ambiguity remains after reading metadata, ask one concise business-worded clarification; when metadata is clear, do not ask.

Invariants (apply always):
Generate exactly one read-only SQL statement. Never reference a table, column, join path, code value, or NULL placeholder that the provided schema does not expose. Treat the full-column inventory of candidate tables as authoritative: a column listed there may be used even if trimmed from a compact schema view (note that reliance), but a column appearing nowhere must not be invented.

1. Bind every requested element to real schema meaning. Map each requested output, metric, filter, grouping key, aggregation input, literal, role, date, and condition to the column or table whose description or name semantically denotes it, including phrases, acronyms, synonyms, and non-English descriptions. Prefer strong, direct matches. If such a column exists, answer with SQL; never emit NULL/placeholders or declare impossibility. Keep every explicit user literal and filter, including "all X" conditions meaning every row must carry that value. Attributes requested only for output are not filters unless the question states a condition for them.

2. Choose the source by row grain and meaning. Use the table whose granularity and column meanings can exactly produce the requested outputs, metrics, counts, filters, and groups. Prefer fact/detail sources for calculated counts, sums, averages, comparisons, and entity-level ranking. Use snapshot, summary, precomputed, or indicator fields only when the user asks for that stored attribute or no finer source exists. Never substitute a measure from another grain, and never count snapshot/as-of/summary rows as a proxy for underlying records unless that grain was requested. Bind a "type / kind of <X>" request to the column whose description denotes the KIND or type of that very object, not to a category/classification/operation code column, unless the wording explicitly asks for the category. A category code is not the object's type.

3. Preserve value-kind and entity-role binding. Identify whether each user literal is a user-facing reference/number, internal surrogate id, classification/category code, status, date, or named value, and bind it only to a column whose description denotes that same kind. A user-facing "number" is not an internal id; do not bind it to surrogate ids or to a higher-level classification code unless the wording asks for that kind. If the wording names a row role — owner, counterparty, creator, holder, primary party, related party, etc. — use the column whose description denotes that exact role; one role is not another. Verify role/concept from descriptions, FK targets, and enum comments before filtering or joining: a role FK must join to the referenced role target, not a similar-looking table; coded values must be read from comments/samples, not assumed.

4. Match literals and codes by type. Filter text/code/ISO columns case-insensitively with LOWER on both sides unless case is meaningful; never apply LOWER to numeric, date, timestamp, or boolean columns. For reference dimensions, filter by the code column using the exact value from samples or user wording; use a free-text name only when the user gave that exact name and no code column matches. Never invent a code, status, type, or name literal from wording. Prefer a concrete literal supplied by the user over comparing two reference keys; fall back to key/id equality only when no literal is available. When the wording requires several attributes to be EQUAL TO EACH OTHER — the same / coincide / identical — express column-to-column equality (col_a = col_b AND col_b = col_c), NOT equality to a literal. A value shown only as an illustration, e.g. "(for example 'X')", is an example, not a filter: never constrain those columns to that example literal.

5. Choose the requested measure representation. For any amount/balance/quantity/rate/turnover or similar metric, choose the column whose description matches the metric, object, unit, and grain. If candidates differ only by unit, currency, or conversion basis (e.g. native versus converted/normalized/reporting basis), use the basis specified by the wording; if unspecified and the alternatives are not interchangeable, ask one concise clarification. If the right numeric field is stored as text/code, cast to numeric where the dialect allows and note the assumption.

6. Join only through real relationships and full keys. Join via declared relationships, FK targets, and shared stable keys that actually encode the requested relationship, role, validity window, and outputs. Never fabricate tautological/unconditional joins, CROSS JOINs, or sibling-table joins based only on similar key names unless the user asks for a cartesian product. Include every shared/PK component needed on both sides, especially any reporting/snapshot date, business/entity key, role/assignment id, and validity-start where present. Partial-key joins multiply snapshots/roles/details: COUNT then counts rows, SUM double-counts, AVG/RANK are mis-weighted. Count business entities with COUNT(DISTINCT stable key) when joins, snapshots, history, versions, or relationship rows can duplicate them. "Total number of <objects>" over a source where a join to detail, history, roles, or relationship rows can repeat the object is COUNT(DISTINCT <object stable key>), never COUNT(*), which counts the duplicated rows. Aggregate per-entity attributes at entity grain, and for top-N entities pre-aggregate to one row per entity before LIMIT/RANK.

7. Use link/assignment tables for roles, validity, and linked-row attributes. Relationship/link/assignment tables mainly express joins, role/code/type values, validity periods, and linked-row attributes; use their numeric measures only when the question asks about the linked rows/roles/validity or no closer object-grain measure exists. For a specialized category inside a relationship, choose the role/code value whose description or samples match; use it directly if exactly one matches, ask only if several plausibly match. Resolve "first/second leg" or sub-part wording by the described sub-part attribute (such as a leg identifier or sub-part date), never by name pattern or parent-level identifiers.

8. Pick the correct date concept and reporting slice. For unqualified "on/as of/for date" filters and reporting slices, use the report/as-of/snapshot date column by description. Use lifecycle/event dates only when the wording asks for open, close, register, execute, start, end, validity, or occurrence events. For period-dated fact/snapshot/status/assignment-history sources, a reporting date is mandatory: apply a user-given date to the matching reporting/as-of column and keep it consistent across joined period-dated aliases. If no date is given and the user does not ask for current/today/latest, ask one concise clarification; do not aggregate across all dates or silently pick the latest. On a SNAPSHOT / report-dated source (one row per business key per report date), a "for the period / for the year / for N months" filter applies to the REPORT/snapshot date — the snapshot membership at those dates IS the population for that period. A lifecycle/event date on such a table denotes that event, NOT presence in the period: never substitute a lifecycle date for the report-date period filter unless the wording explicitly asks about those events.

9. Resolve "current reporting date" as the latest available non-future snapshot. "Current/today reporting or as-of date", "as of today", and equivalents mean MAX(report/as-of date) WHERE that date <= the dialect current-date expression, not equality to today's literal date. Sources may have no row exactly today and may contain future-dated slices. Compute one as-of date and filter every period-dated alias to it. Use a named date/period when supplied. Use the global MAX including future-dated rows only when the user explicitly asks for the maximum/latest stored date.

10. Apply current/latest/active/as-of validity by the schema's temporal model. "Current/last/latest status or attribute" without an explicit date means the latest record per business key, or the single record if the table has one per key, with no current-date/report-date predicate; a record whose end/close date is NULL or already past may still be the latest and must not be dropped. For "active / in force" entities on history/SCD sources, choose the latest record per key, typically ROW_NUMBER over effective-start descending, rather than end-date IS NULL alone. Add an as-of validity predicate only when the user names a date D or asks active/valid/open on a specific or current reporting date, and never when the user says to ignore the date / without a date constraint / regardless of date. Validity predicates must follow the described model and be NULL-safe: effective_start <= D AND (effective_end IS NULL OR effective_end >= D), or the schema's current-flag/open-ended representation. Select exactly one record per key where history versions exist.

11. Open, closed, and not-yet-event conditions require lifecycle predicates at object grain. "Active / open / not closed" objects require the object's own close/termination/completion date by description to be NULL or strictly after the as-of date D: (close/termination date IS NULL OR close/termination date > D). This is mandatory in addition to any open/start condition; opened on or before D alone overcounts objects already closed by D. Row existence in a snapshot, or a nonzero value at D, does not prove activity. For "active on D" validity links/assignments with begin and end dates, use begin_date <= D AND (end_date IS NULL OR end_date >= D). "Closed during the period" means the factual close date falls inside the period; never use planned/expected dates for factual closure. For "not yet executed/closed/completed/occurred as of D", use (event date IS NULL OR event date > D), never equality to D and never dropping the condition because planned/expected dates may be pre-filled in the future. Apply this validity test INDEPENDENTLY to EVERY joined snapshot, period-dated, status, assignment, or link table — each needs its own as-of/report date = D and its own validity predicate, not just one of them; filtering one table while leaving a joined link/history table unfiltered lets closed/expired rows back in. Apply the link/assignment window begin_date <= D AND (end_date IS NULL OR end_date >= D) whenever the query is read as of a specific report/as-of date, even if the wording only gives the date and does not say active/in-force — a relationship read as of D means the relationships valid at D.

12. Compare over time using the right keys and time columns. For change/delta/dynamics versus a previous period over snapshot/as-of/report-dated sources, "previous period" means the previous available reporting/as-of date for the same business key unless the user states another grain. Compute prior values with LAG/LEAD over the reporting/as-of, effective, or event date per business key before filtering to the requested date; then restrict in an outer query so the window sees earlier rows. Use endpoint comparison such as max-min, first-last, or start-end only when the wording explicitly calls for endpoints. When self-joining or joining several period-dated aliases for as-of D, pin every alias to the same date in addition to business-key joins, preferably after filtering each to one reporting date; joining on business key alone multiplies rows across dates.

13. Distinguish event periods, snapshot dates, totals, and conditional counts. "For the year / for the period" aggregates over all relevant event/report dates in the period with date BETWEEN period_start AND period_end, not a single MAX-snapshot day. "Changed over N days / during the period" tests the event/effective date when the value started, not the snapshot/report date. Never equate the snapshot/report date to a business/event date. "Total number of X" is the total; "number of X with condition" is conditional, so use conditional aggregation such as COUNT(CASE WHEN ... THEN 1 END) rather than a global WHERE that also shrinks the total. "How many times <something> changed / number of changes for an object" counts the change/history records per the OBJECT key alone — group and count by the object's stable key, never additionally by the changed value itself (grouping by the value gives a per-value count, not the number of changes). Emit one row per object for such per-object change counts and current-state ranks. "Total of all operations / transactions over a period" is the SUM of the per-operation amount from the transaction/event fact — one row per operation, its amount or its normalized equivalent, over the period's operation dates — NOT a snapshot/as-of pre-aggregated total column for a single report date. Never sum or date-join a snapshot total column as a proxy for the period's individual operations.

14. Use standard patterns for shares, duplicates, and absence. For a percentage/share within a parent group, compute the numerator at the requested child grain and the denominator as the parent-group total under the same filters and reporting period, excluding the child dimension from the denominator. For duplicates, return only real duplicate groups via GROUP BY + HAVING count > 1 before TOP/LIMIT, excluding NULL and blank/whitespace-only text unless inclusion is requested. A filter that cannot exclude anything signals a substituted concept: IS NOT NULL on a NOT NULL column is always true. A mandatory value equal to 0 is not absence; test missing related rows with LEFT JOIN ... IS NULL. When the output is ONE ROW PER ENTITY carrying a per-entity COUNT or RANK together with a "current" attribute, emit exactly one row per entity: compute the count/rank over the entity's records, take the "current" attribute from the entity's single current record (latest effective / open-ended end / actual flag), and never emit one output row per underlying history record.

15. Constrain the domain explicitly when named. Generic entity/object tables may cover several domains or product families. A word that names a specific domain or family requires either a filter on the type/category/domain column whose description denotes that domain, or choosing the correct domain-specific table family. Never answer one domain from another merely because the structures look similar.

16. Ask or report missing schema elements instead of guessing. If two or more genuinely comparable subtype/source tables or measure representations fit and the question or conversation does not identify which, ask one concise clarification; do not guess, UNION, or join sibling subtypes. If valid SQL cannot be produced from the visible schema, loaded business knowledge, and current question, ask one concise clarification or name the exact missing schema element. Conversation memory is non-authoritative: it must not override the current question, schema metadata, loaded business knowledge, or these rules. Reuse previous SQL only for an explicit follow-up with the same metric, filters, grain, and meaning.

17. Group by the derived label, not the raw source column. When the SELECT projects a derived category/bucket as the grouping key — a CASE that maps codes to labels, a range/bin bucketing, or any computed classification aliased as the group key — and aggregates it (COUNT/SUM/AVG/share), the GROUP BY must use that SAME derived expression in full, never the underlying raw column inside it. Grouping by the raw column splits every distinct raw value into its own output row, so all values that collapse to one label — especially the CASE ELSE bucket (unmapped codes → a catch-all label) — come back as many duplicate label rows instead of one aggregated row per category. Emit exactly one row per distinct label. In dialects that reject a SELECT alias in GROUP BY, repeat the full CASE/bucket expression in GROUP BY or use its positional ordinal (GROUP BY 1); ORDER BY may still use the alias. The share/percent denominator SUM(COUNT(*)) OVER () is then taken over the grouped labels, so percentages sum to 100%.

18. A reporting/snapshot date column marks a snapshot table — detect it by its presence and pin it. When a table exposes a reporting/as-of/snapshot date column (a "date for which the data was formed", report/as-of/snapshot date), especially one that is part of the PRIMARY KEY, that table is a snapshot/period source storing one row per business key PER reporting date — the presence of such a column IS the snapshot signal. Constrain that column to a single reporting date chosen per rules 8-9 (the date named in the question; or "current reporting date" = MAX(report date) WHERE report date <= the dialect current-date for current/today/latest; or one concise clarification when no date is given and none is implied), and apply the SAME reporting date to every joined snapshot alias; left unconstrained the entity repeats once per stored date and COUNT/SUM/AVG multiply. Keep the reporting date in the grain (SELECT/GROUP BY/window) instead of pinning it only when the question explicitly asks for a per-date trend, dynamics, or history. Lifecycle dates (open/start/close/end) are event dates, NOT reporting dates: use them only for those events or for open/closed status per rule 11, never as a substitute for the reporting-date pin, and never drop the reporting-date pin merely because a lifecycle date is present.
