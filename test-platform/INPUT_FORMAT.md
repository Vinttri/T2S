# Как прислать данные для нового бенчмарка (по-человечески)

Это инструкция «что кидать», чтобы я завёл новый бенчмарк и показал результаты на дашборде.
Без жаргона — простыми словами.

## Как вообще считается «правильно/неправильно»

Мы не сравниваем тексты ответов. Мы **берём SQL, выполняем его на базе и сравниваем
получившиеся таблицы**. Поэтому «правильный ответ» к вопросу — это **эталонный SQL**
(мы его зовём *gold*), а не словесный ответ и не «вот число 42».

Из этого следует простая вещь: чтобы что-то проверить, мне нужны **две** вещи —
1. **сама база данных** (на чём выполнять), и
2. **список вопросов, и к каждому — эталонный SQL**.

## Что присылать

### 1. Базу данных
Postgres-дамп (`pg_dump`), или SQL-скрипт, который её создаёт, или просто скажи откуда её
взять. Если это известный датасет (LiveSQLBench/BIRD/Spider и т.п.) — достаточно названия,
у нас уже есть откуда грузить. Без базы бенчмарк не запустить.

### 2. Вопросы и эталоны — лучше всего одним файлом JSONL
JSONL = текстовый файл, **по одному JSON-объекту в строке**. Один объект = один вопрос.
Вот всё, что в нём бывает (жирным — обязательное):

```jsonc
{
  "case_id":  "myds_1",                 // **уникальный id вопроса** (любой, лишь бы не повторялся)
  "db_id":    "my_dataset",             // **имя базы** (как она называется в Postgres)
  "question": "Топ-5 продавцов по выручке за последний квартал?",  // **сам вопрос словами**
  "gold_sql": "SELECT ... LIMIT 5;",    // **эталонный SQL — правильный ответ**

  "conditions":   { "decimal": 2, "order": true, "distinct": false }, // как сверять (см. ниже)
  "difficulty":   "Hard",               // сложность; не пришлёшь — посчитаю сам
  "normal_phrasing": "...",             // «причёсанная» формулировка вопроса (если есть)
  "category":     "Query"                // тип; берём только SELECT-вопросы (Query)
}
```

Минимум, если совсем лень: **`case_id` + `question` + `gold_sql`** (плюс база). Остальное
подставлю по умолчанию.

### Что значит `conditions` (как сравнивать таблицы)
- **`order`** — важен ли порядок строк. `true`, если в вопросе есть «отсортируй / топ / по
  убыванию» и порядок надо проверять. Иначе сравниваю как множество (порядок не важен).
- **`decimal`** — до скольких знаков округлять числа перед сравнением. `-1` = не округлять.
- **`distinct`** — учитывать ли уникальность строк.

Не уверен в `conditions` — ставь `{ "decimal": -1, "order": false, "distinct": false }`,
это безопасный дефолт.

## Пример (3 строки JSONL)
```json
{"case_id":"shop_1","db_id":"shop","question":"Сколько всего заказов?","gold_sql":"SELECT COUNT(*) FROM orders;","conditions":{"decimal":-1,"order":false,"distinct":false}}
{"case_id":"shop_2","db_id":"shop","question":"Топ-3 товара по выручке, дороже всего сверху.","gold_sql":"SELECT name, SUM(price*qty) rev FROM items GROUP BY name ORDER BY rev DESC LIMIT 3;","conditions":{"decimal":2,"order":true,"distinct":false}}
{"case_id":"shop_3","db_id":"shop","question":"Список уникальных городов покупателей.","gold_sql":"SELECT DISTINCT city FROM customers;","conditions":{"decimal":-1,"order":false,"distinct":true}}
```

## Важное правило
Каждый `gold_sql` **должен выполняться на присланной базе без ошибок**. Я это автоматически
прогоняю перед тем, как завести бенчмарк, и если эталон падает — верну на доработку
(с указанием, какой именно вопрос сломался).

## Частые «а если…»
- **Нет SQL, есть только ожидаемая таблица результата.** Тогда вместо `gold_sql` пришли
  `expected_result: {"columns": [...], "rows": [[...], ...]}`. Это другой режим сверки —
  предупреди заранее, подкручу.
- **Ответ — просто число/текст, без SQL и без таблицы.** Так не получится посчитать
  execution-match; сделаем отдельный режим (LLM-судья), но это надо обсудить.
- **Хочешь добавить не датасет, а новую систему-участника** (которую бенчмаркаем) — это
  отдельная история, см. `CLAUDE.md` → «HOW TO: add a participant».

## Что будет дальше (после того как пришлёшь)
1. Залью базу в наш Postgres.
2. Сохраню твой JSONL как `BENCHMARK_<db_id>.jsonl` и проверю все эталоны на исполнение.
3. Пропишу `bench.<db_id>.json` и прогоню участников.
4. Результаты сольются в дашборд — выберешь бенчмарк в селекторе вверху.
5. Забрать результаты машинно можно ручкой `…/evaluation-dashboard-api/results` (JSON по всем
   бенчам) или кнопкой ⬇ на странице. Подробности — в `CLAUDE.md` → «Dashboard & hosting».
