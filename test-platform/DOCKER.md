# Docker Deployment For `bench_app`

This packages the current live benchmark app as a split Docker Compose stack:
backend is FastAPI (`bench_app.server:app`) on the internal compose port
`8090`, worker executes durable benchmark jobs from `/data/app.db`, backup
periodically copies runtime state to `/backups`, and frontend is nginx on
internal port `8080` serving static UI assets and proxying `/api` and `/ws` to
the backend.

Target deployment for this package is Docker Compose. Kubernetes manifests are
not required for the isolated-contour deployment path.

## Build

```bash
cd /root/leaderboard_builder_codex
scripts/build-bench-app-docker.sh
```

Custom image/tag:

```bash
scripts/build-bench-app-docker.sh \
  --image my-registry/leaderboard-bench-app \
  --frontend-image my-registry/leaderboard-bench-frontend \
  --tag 20260605
```

Build inside an isolated network with an internal PyPI mirror:

```bash
scripts/build-bench-app-docker.sh \
  --image leaderboard-bench-app \
  --tag offline \
  --pip-index http://pypi.local/simple \
  --pip-trusted pypi.local
```

For a fully offline source build, the Docker host also needs the base images
`python:3.12-slim` and `nginx:1.27-alpine` already loaded or available from an
internal registry. The offline bundle generated by
`scripts/export-bench-app-offline.sh` already contains built backend and
frontend images. The worker and backup services use the backend image.

## Run

```bash
mkdir -p data reviews backups
docker compose up -d
```

Open `http://127.0.0.1:8090/`.

For host folders instead of Docker named volumes:

```bash
mkdir -p /opt/leaderboard/data /opt/leaderboard/reviews /opt/leaderboard/backups
# edit these in .env:
# BENCH_DATA_DIR=/opt/leaderboard/data
# BENCH_REVIEWS_DIR=/opt/leaderboard/reviews
# BENCH_BACKUPS_DIR=/opt/leaderboard/backups
docker compose up -d
```

All runtime state lives in mounted folders. The image contains code and static
assets only, not `bench_app/data`.

## Run With Docker Compose

The repository includes `docker-compose.yml` for a four-container deployment:

- `backend`: FastAPI/uvicorn, no public port, persistent `/data` and `/reviews`.
- `worker`: benchmark executor, persistent `/data` and `/reviews`, no public
  port, no backend HTTP healthcheck.
- `backup`: periodic SQLite + runtime artifact backup, persistent `/data`,
  `/reviews`, and `/backups`, no HTTP healthcheck.
- `frontend`: nginx static UI, public `BENCH_HOST_PORT`, proxies `/api` and
  `/ws` to `backend:8090`.

The compose file uses `restart: "no"` for app containers so crashes stay
visible in `docker compose ps` and logs instead of being hidden by restart
loops. It never pulls from the internet (`pull_policy: never`), so load both
images first:

```bash
docker load -i leaderboard-bench-app_env_mount_image.tar
```

Prepare runtime folders and env:

```bash
mkdir -p data reviews backups
cp .env.example .env
```

Edit `.env`, then start:

```bash
docker compose up -d
```

Open `http://127.0.0.1:8090/`.

By default Compose mounts:

```text
./data    -> backend+worker /data
./reviews -> backend+worker /reviews
./backups -> backup /backups
```

To use another host port or host folders:

```bash
# edit these in .env:
# BENCH_HOST_PORT=8085
# BENCH_DATA_DIR=/opt/leaderboard/data
# BENCH_REVIEWS_DIR=/opt/leaderboard/reviews
# BENCH_BACKUPS_DIR=/opt/leaderboard/backups
# BENCH_BACKEND_IMAGE=leaderboard-bench-app:latest
# BENCH_FRONTEND_IMAGE=leaderboard-bench-frontend:latest
docker compose up -d
```

## Runtime Configuration

Useful environment variables:

- `PORT`: container listen port, default `8090`.
- `BENCH_STORE_URL`: app runtime store URL, default
  `sqlite:////data/app.db`.
  Supported values are `sqlite:///...` and `postgresql://...`.
- `BENCH_APP_DATA_DIR`: runtime data root, default `/data`.
- `BENCH_APP_RUNS_DIR`: final result JSON dir, default `/data/runs`.
- `BENCH_APP_ANSWERS_DIR`: raw answer JSON dir, default `/data/answers`.
- `BENCH_APP_JUDGED_DIR`: LLM judge JSON dir, default `/data/judged`.
- `BENCH_APP_RUN_LOGS_DIR`: per-run JSONL logs dir, default `/data/logs`.
- `BENCH_APP_STDOUT_RUN_LOGS`: mirror per-run JSONL events to container stdout
  for `docker logs`, default `1`.
- `BENCH_APP_STDOUT_RUN_LOG_MAX_CHARS`: max characters per mirrored stdout log
  line before truncation, default `20000`.
- `BENCH_APP_DATASETS_DIR`: uploaded benchmark JSONL dir, default
  `/data/datasets`.
- `BENCH_APP_CONNECTORS_YAML_DIR`: connector YAML mirror dir, default
  `/data/connectors`.
- `BENCH_APP_REVIEWS_DIR`: editable solution-review markdown dir, default
  `/reviews`.
- `BENCH_APP_SYNC_CONNECTOR_YAML`: mirror connectors to YAML, default `1`.
- `BENCH_APP_RUNNER_MODE`: `worker` for Docker Compose. In this mode backend
  enqueues jobs and the worker executes long benchmark work.
- `BENCH_APP_WS_SNAPSHOT_INTERVAL_S`: interval for compact DB snapshots over
  WebSocket; default `2` seconds in compose.
- `BENCH_APP_GOLD_CACHE`: cache ideal `gold_sql` query results in process
  memory and prewarm missing answers at run start, default `1`.
- `BENCH_APP_GOLD_CACHE_CONCURRENCY`: max concurrent DB queries during gold
  cache prewarm, default `4`.
- `BENCH_APP_MAX_API_CONCURRENCY`: max connector/API questions waiting on model
  responses across all active runs, default `1`.
- `BENCH_APP_MAX_IMPALA_CONCURRENCY`: max active Impala scoring SQL executions
  across active runs and manual SQL checks, default `1`.
- `BENCH_APP_CIRCUIT_BREAKER_ENABLED`: pause a run after repeated external
  API/DB/LLM transport failures, default `1`.
- `BENCH_APP_CIRCUIT_BREAKER_FAILURES`: default consecutive failure threshold,
  default `5`. Per-kind overrides: `BENCH_APP_CIRCUIT_BREAKER_API_FAILURES`,
  `BENCH_APP_CIRCUIT_BREAKER_DB_FAILURES`, `BENCH_APP_CIRCUIT_BREAKER_LLM_FAILURES`.
- `BENCH_WORKER_STALE_AFTER_S`: stale worker heartbeat age before a running job
  is recovered, default `900`.
- `BENCH_WORKER_WATCHDOG_INTERVAL_S`: periodic stale-job watchdog interval,
  default `30`.
- `BENCH_WORKER_MAX_JOB_ATTEMPTS`: stale retries before a job/run is failed,
  default `3`.
- `BENCH_BACKUP_ENABLED`: run periodic on-disk backups, default `1`.
- `BENCH_BACKUP_INTERVAL_S`: backup interval in seconds, default `1800`.
- `BENCH_BACKUP_KEEP`: backup files retained per backup class, default `48`.
- `BENCH_BACKUP_DIR`: container backup directory, default `/backups`.
- `BENCH_BACKEND_MEM_LIMIT`, `BENCH_WORKER_MEM_LIMIT`,
  `BENCH_FRONTEND_MEM_LIMIT`, `BENCH_BACKUP_MEM_LIMIT`: compose memory caps.
- `BENCH_BACKEND_CPUS`, `BENCH_WORKER_CPUS`, `BENCH_FRONTEND_CPUS`,
  `BENCH_BACKUP_CPUS`: compose CPU caps.
- `BENCH_APP_AUTO_JUDGE`: run LLM L0-L4 judge automatically, default `1`.
- `LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL`: default LLM judge credentials.
- `LLM_AUTH_HEADER`, `LLM_AUTH_SCHEME`: LLM auth transport. Default is
  `Authorization: Bearer <LLM_API_KEY>`. Set `LLM_AUTH_SCHEME=none` to send a
  raw key, for example through `X-API-Key`. Set `LLM_AUTH_HEADER=none` for a
  no-auth internal gateway.
- `LLM_JUDGE_TIMEOUT`, `LLM_TEST_TIMEOUT`: judge and Settings-tab test request
  timeouts. For large internal models, use values such as `3600`.
- `LLM_JUDGE_CONCURRENCY`: max LLM-evaluated questions across active benchmark
  runs, default `1`.
- `LLM_JUDGE_MAX_RETRIES`, `LLM_JUDGE_RETRY_DELAY`: automatic retries for
  failed/invalid LLM judge attempts.
- `BENCH_APP_SSL_VERIFY`: default/sample is `0` for internal contours with
  self-signed TLS. Set `1` only when LLM/connectors/middleware must verify
  public CA certificates.
- `BENCH_DM_MIS_IMPALA_DSN`: scoring DB connection string for the default
  seeded `dm_mis` Impala datasets (`1`, `3`, `10`, and `54` questions).
  `impala://...` DSNs are supported through `impyla`; use query params such as
  `auth_mechanism=LDAP`, `use_ssl=true`, `verify_cert=false`,
  `request_pool=...`, and `connect_timeout=10`.
- `BENCH_APP_DB_SSL_VERIFY`: optional fallback for Impala certificate
  verification when the DSN does not contain `verify_cert=...`; default is off.

Example with judge credentials:

```bash
cp .env.example .env
# edit LLM_BASE_URL / LLM_API_KEY / LLM_MODEL in .env, then:
docker compose up -d
```

## Scoring Database

Bundled and seeded datasets point at scoring DB DSNs like
`postgresql://bank:bankpass@localhost:15432/<db_id>`. Inside Docker,
`localhost` means the container itself. For real benchmark runs in a new
environment, either:

- use `host.docker.internal` in the scoring DSN if the scoring DB is on the
  Docker host;
- set dataset DSN env vars such as `BENCH_<DB_ID>_<DB_TYPE>_DSN`,
  `BENCH_<DB_ID>_DSN`, or `BENCH_SCORING_DSN` to a reachable DB host;
- pass `BENCH_DM_MIS_IMPALA_DSN=...` for the seeded `dm_mis` Impala datasets;
- provide a migrated `app.db` through the mounted data volume.

Do not push images containing production connector configs or SQLite data to a
public registry unless secrets have been removed.

## Mounted Data Layout

Recommended bind mounts:

```text
/opt/leaderboard/data    -> /data
/opt/leaderboard/reviews -> /reviews
/opt/leaderboard/backups -> /backups
```

The app writes:

```text
/data/app.db                 SQLite runtime store
/data/connectors/*.yaml      connector mirror, if enabled
/data/answers/*.json         raw connector answers
/data/judged/*.levels.json   LLM L0-L4 judge outputs
/data/runs/*.json            final run results
/data/logs/*.jsonl           append-only per-run logs
/data/datasets/*.jsonl       uploaded benchmark files
/reviews/*.md                editable solution reviews
/backups/app-db/*.db         periodic SQLite backups
/backups/data/*.tar.gz       periodic runtime artifact backups
```

## Offline Bundle

For an isolated network with no internet access, build the image and export it
as a portable archive on a machine that has Docker build access:

```bash
cd /root/leaderboard_builder_codex
scripts/export-bench-app-offline.sh --tag offline
```

If exporting from inside a contour that uses internal PyPI:

```bash
scripts/export-bench-app-offline.sh \
  --tag offline \
  --pip-index http://pypi.local/simple \
  --pip-trusted pypi.local
```

Copy the generated archive from `dist/offline/` into the isolated network:

```bash
tar -xzf leaderboard-bench-app_offline.tar.gz
cd leaderboard-bench-app_offline
cp .env.example .env
./run-bench-app-offline.sh
```

This loads both Docker images from `image.tar.gz` if needed and starts the app
with Docker Compose. No Python package download is needed inside the isolated
network.

The offline bundle contains `.env.example` and `docker-compose.yml`. Copy it to
`.env` to set Docker launch options, the app runtime DB (`BENCH_STORE_URL`),
and your internal OpenAI-compatible LLM judge (`LLM_BASE_URL`, `LLM_API_KEY`,
`LLM_MODEL`, `LLM_JUDGE_MAX_RETRIES`, `LLM_JUDGE_RETRY_DELAY`). The offline
runner passes only `.env` to Docker Compose by default; ambient shell variables
are not forwarded.
